# Spec Workflow

> 你有一个想法。你的 AI Agent 帮你把它变成一份程序员拿到就能开工的产品规格书。

---

## 这解决什么问题？

你一定经历过：

> 刷到一篇文章，看到一个产品，脑子里闪过"这个能做"。
> 然后你开始想：市场多大？技术上行不行？分几步做？先做什么？
> 你想写个文档把思路理清楚，但写着写着就乱了——不知道写多细、不知道缺什么、不知道这个东西到底能不能落地。

**Spec Workflow 就是干这个的。**

它是一套你和 AI Agent 协作的标准流程。你只管说"研究一下这个想法"——Agent 自动走完：

```
调研 → 出骨架给你看方向 → 出完整 Spec → 送专家评审 → 
改到没问题 → 定名字 → 冻结 → 程序员直接开工
```

整个过程你用日常对话参与，不用写一行文档。

---

## 为什么你需要它？

| 没有它 | 有了它 |
|--------|--------|
| 想法在脑子里，说不清楚 | 30 分钟出一份 400 行的方向骨架 |
| Spec 写了一堆，程序员说"看不懂" | 每个字段、每个命令、每个状态都有，无歧义 |
| 不知道做到什么程度算完 | 有评分体系（8.5+），有验收清单 |
| 一个人拍脑袋，不知道对不对 | 自动送 coding agent + 行业专家交叉评审 |
| 改了 5 版还是觉得不对 | 每轮只改当前发现的问题，改完立刻送审 |

## 安装

```bash
clawhub install spec-workflow
```

或者：

```
对着你的 AI Agent 说：安装 skill https://github.com/midsummergit/spec-workflow
```

安装后，任何时候说**"研究一下 XX"**或**"写 Spec"**，Agent 自动进入工作流。

## 怎么用？

```bash
# 第一步：你有个想法
你: "研究一下 AI 短视频素材拆解工具"

# 第二步：Agent 自动调研，然后问你要个临时代号
Agent: "调研完了。先定个代号？正式名后面再想。"
你: "先叫 素材捕手"

# 第三步：Agent 出 V0.1 骨架（~400 行）
Agent: "方向草稿好了。分三个阶段：个人工具 → 团队版 → 平台。你看看方向对不对？"
你: "方向对，继续"

# 第四步：Agent 出完整 Spec，送审
Agent: "V1.0 写完了。Codex 评分 7.4，专家提了 8 条建议。"
Agent: "6 条我建议改，2 条我建议不改——这是 P0 不需要的。你看呢？"
你: "同意，改那 6 条"

# 第五步：迭代到通过
Agent: "V2.8，Codex 8.6 分，专家说可以冻结了。要不要定正式名？"
你: "叫 灵感捕手，英文 hookly，Slogan：网络爆款，帮你抓得住，拆得透，做得出"
Agent: "已全局替换。检查无残留。"

# 第六步：冻结
你: "冻结"
Agent: "已打 tag frozen-v2.8。程序员读 Part 0 就可以开工了。"
```

## 这个 Skill 里有什么？

| 内容 | 说明 |
|------|------|
| **SKILL.md** | Agent 加载后立即进入专业状态（触发条件、流程速查、质量规则、反模式） |
| **PLAYBOOK.md** | 完整参考手册（文档模板、28 项 artifact 清单、一致性检查清单、Git 版本管理） |
| **examples/hookly-spec.md** | 灵感捕手完整 Spec（1890 行，实操参考） |

## 沉淀自真实项目

这套流程不是凭空设计的。它从**灵感捕手 (hookly)** 项目里长出来的：

- 从一篇微信公众号文章出发
- 经过 20+ 轮迭代
- 经 Codex 和行业专家 8 轮交叉评审
- 最终产出一份 1900 行、程序员可直接开工的产品 Spec

---

## 适合谁？

- **独立开发者 / 创业者**：想法多，需要一个结构化的方式把想法变成可执行的方案
- **产品经理**：需要快速产出高质量 PRD，且经得起技术评审
- **技术负责人**：需要确保 Spec 粒度够细，程序员不需要来问
