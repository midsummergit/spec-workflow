# Product Spec Playbook — 从 Idea 到可开工 Spec

> 沉淀自灵感捕手 (hookly) 项目。适用于任何"我有一个想法→做出可开工的产品规格书"的场景。
>
> **核心理念**：好的 spec 不是写出来的，是审出来的。程序员拿到就能搭，不需要问。

### 快速启动

```
1. 决策者: "研究一下 XXX"
2. Agent:  调研 → 定 @PROJECT@（项目中文名） / @ENG@（项目英文名） / @SLOGAN@（一句话 Slogan）
3. Agent:  出 V0.1 骨架（~400 行，三阶段框架）→ 决策者确认方向
4. Agent:  出 V1.0 全量 → 送 coding agent + 专家评审
5. 多轮迭代 → 评分 ≥8.5 → 决策者定品牌名
6. sed 一键替换 placeholder → 决策者确认冻结
7. 程序员读 Part 0 开工
```

---


## 流程总览

```
Idea → 调研 → V0.1 骨架 → 方向确认 → V1.0 全量 → 外部评审 → 迭代闭合 → 品牌化 → 冻结交付
         ↓          ↓           ↓           ↓           ↓           ↓          ↓         ↓
      1 小时     30 分钟     30 分钟     2 小时      30 分钟     N×30 分钟   30 分钟   10 分钟
```

---


## 协作心态

> 从调研立项到 Spec 冻结，是一个多轮迭代的过程。保持耐心，每轮尽职尽责做到最好。

- **预期多轮迭代**：hookly 从 V1.0 到 V2.8 经历 20+ 轮。不是一次写对的，是一轮轮审出来的。
- **该研究就研究**：不确定的技术方案，花时间验证，不在 Spec 里猜。
- **该收集数据就收集数据**：不能拍脑袋的决策（定价、成功率 SLA），标注 `[待验证]` 并给出验证路径。
- **Agent 可以建议"这个版本可以开工了"**——但最终冻结决定权在决策者。
- **每轮交互 10-30 分钟**：快节奏推进，不陷入完美主义。
- **不要跳过评审**：自己写的 Spec 一定有盲区。外部视角（coding agent + 领域专家）是质量保障。

## 版本管理

> Spec 文档用 Git 管理，不靠文件名或文档内 changelog 追历史。

### Git 工作流

```
每次重大修改 → git commit
评审通过 → git tag -a v2.8 -m "coding-agent 8.6, expert approved"
冻结版 → git tag -a frozen-v2.8
```

### Commit 规范

```
hookly V2.3: 反馈有机融入 — 反馈拆入各 Part 执行流程/交互/API
hookly V2.4: 品牌定名 — 灵感捕手 + hookly, 全文统一
hookly V2.8: 开工前终版 — 13项一致性修复, coding-agent 8.6/8.8 READY
```

> **格式**：`项目名 版本号: 一句话改动 — 关键细节`
> **原则**：每轮评审反馈改完后立即 commit。不要攒几轮一起交。

### Tag 规范

| Tag 类型 | 格式 | 何时打 |
|---------|------|--------|
| 评审版 | `v2.4` | 每次送 coding agent / 专家评审的版本 |
| 冻结版 | `frozen-v2.8` | 决策者确认可以开工的版本 |
| 里程碑 | `v1.0-first-draft` | 初稿完成 |

### 分支策略

```
main ─── 始终是当前最新稳定版
  │
  ├── v1.x  V1.0-V1.4 的迭代（初稿→重定位→品牌化前）
  ├── v2.x  V2.0-V2.8 的迭代（架构重构→终版）
  │
  └── 不另开 feature 分支（Spec 是单人编辑，不需要 PR 流程）
```

> **原则**：Spec 迭代快（每 10-30 分钟一版），开分支反而增加合并摩擦。直接 main 上 commit + tag 即可。

### 文件命名

```
PROJECT-spec.md        # 全量产品规格书（唯一 spec 文件）
PROJECT-coding-review-v1.md   # Coding agent 评审报告（每轮一个文件）
PROJECT-expert-review.md     # 专家评审记录
```

> **原则**：一个项目只有一个 spec 文件。不要拆成 P0/P1/P2 多个文件（维护噩梦）。
> **例外**：评审报告独立存放，便于追溯评审历史。

### 不要做的事

- ❌ 用 `spec-v1.md`, `spec-v2.md`, `spec-final.md`, `spec-final-v2.md` 管理版本
- ❌ 把 spec 拆成 `P0_SPEC.md`, `P1_SPEC.md`, `P2_SPEC.md`
- ❌ 不改 Changelog 只靠 git log 追历史（Changelog 给人类看，git log 给机器看，两者都要）

## Phase 1: 发现与调研

**输入**：一个想法（一篇文章、一个竞品、一个痛点）

**步骤**：
1. 搜索原始来源（公众号/博客/视频）
2. 找到竞品/类似系统
3. 了解技术栈（用什么工具、什么 API）
4. 判断商业可行性（谁在用、付费吗、天花板多大）

**产出**：调研笔记（来源链接、竞品列表、技术可行性判断）

**质量门**：能回答"这个东西有市场吗？技术上能做吗？"

---

## Phase 2: V0.1 骨架（方向验证）

**输入**：调研笔记 + 决策者确认的代称

**产出**：骨架 Spec（~400 行），覆盖**所有阶段的主体框架**：
- §A 项目背景与价值（What/Why，含举例）
- §B 增量交付路线总览（P0/P1/P2 三阶段表）
- Part 0 基本大纲（定位、字段名列表、技术栈、执行顺序）
- Part 1 基本大纲（相对于 P0 要加什么、为什么）
- Part 2 基本大纲（相对于 P1 要加什么、为什么）

**目标**：三阶段的大方向一次审完。不让决策者读 1250 行来判断"分期对不对"。

**不写**：详细 DDL、完整 JSON Schema、Prompt 全文、API 命令、错误处理细节。这些在方向确认后才填充。

**质量门**：决策者确认"三阶段划分合理，方向正确，可以写全量"

---

## Phase 3: V1.0 全量

**前置条件**：V0.1 方向已确认

**产出**：全量 Spec 初稿，覆盖所有模块

**原则**：
- 先写全量，再砍范围
- 所有不确定的地方标 `[待验证]`
- 初稿不求完美，求覆盖

**质量门**：≥80% 的模块有描述，Gap 清单≤10 项

---

## Phase 4: 外部评审（第一轮）

**输入**：V1.0 Spec

**步骤**：
1. 送 coding agent review（如 Codex、Claude Code，偏工程规范评审）
2. 送领域专家 review（偏实战可行性）
3. Agent 列出所有评审意见，每条标注：建议采纳 / 建议不采纳 / 需讨论 + 理由
4. 决策者逐条确认 → 形成最终修改计划

**关键原则**：Agent 整理意见、给出专业判断，但**最终决定权在决策者**。不是所有评审意见都要采纳——coding agent 容易过度工程化，专家建议也可能不适配当前阶段。

**Coding agent 评审维度**：
- 商业价值与定位
- 技术架构
- 数据模型
- AI/Prompt 设计
- 接口规格
- 状态机与工作流
- 实施路线图
- 完整性

**专家评审关注点**：
- P0 范围是否合理（有没有膨胀）
- 最大风险点在哪（什么最可能卡死）
- 执行顺序对不对（应该先做什么）
- 哪些过度工程了

**产出**：评审报告 + 修改计划 + 决策记录

**质量门**：评分≥7.0 / 10

---

## Phase 5: 迭代闭合

**输入**：评审报告 + 修改计划

**核心原则**：
- 每轮只做一致性修复，不扩功能
- 改完就送审，直到评审通过
- 决策者确认后才进入下一轮

**常见迭代点**：
1. 重定位（从大系统蓝图→可落地的 MVP）
2. 按增量阶段拆分（P0/P1/P2）
3. 补全 artifact（流程图/DDL/Schema/Prompt/API/Contract/验收）
4. 修复一致性（字段计数/SLA 数字/路径/交互描述）
5. Schema 对齐（Prompt 输出 ⊇ JSON Schema ⊇ 数据模型 ⊇ Build Contract）

**迭代终止条件**：
- Coding agent 评分 ≥8.5
- 专家表示"可以冻结了"
- 剩余问题全是"nice to have"不是"blocker"

---

## Phase 6: 品牌化

**时机**：Spec 内容稳定后（不要过早，会分散注意力）

**步骤**：
1. 明确核心卖点（这个产品最大的差异化是什么）
2. 从卖点出发想名字（中文 + 英文工程名）
3. 定 Slogan（一句话解释产品价值）
4. 更新 Spec 全文统一品牌名

**命名原则**：
- 从用户视角出发，不从技术视角
- 短、好记、有画面感
- 英文名考虑做 repo/package 名

**产出**：产品名 + 工程名 + Slog
## Phase 7: 冻结与交付

**前置条件**：
- 评审通过（coding agent ≥8.5 + 专家认可）
- 所有 artifact 完整
- 一致性检查全部通过

**交付物**：
1. `PROJECT-spec.md` — 全量产品规格书（git tag 标记版本）
2. `git tag frozen-vX.X` — 冻结版标签
3. 可抽取 P0 独立文档（P0 开发人员只需读 Part 0）

**P0 就绪检查**：
- [ ] 字段定义无歧义（类型/枚举/必填）
- [ ] Prompt 有完整 JSON Schema
- [ ] API 命令可直接执行
- [ ] 状态机有图 + 转换规则
- [ ] 错误处理覆盖主路径 + 降级
- [ ] Day 0 smoke test 可执行
- [ ] Build Contract 完整（字段映射 + fixture + smoke test）
- [ ] 程序员只读 Part 0 就能开工

| 角色 | 职责 | 在 hookly 中 |
|------|------|------------|
| **决策者** | 发现机会、定方向、做最终判断、验收质量 | Charles/超哥 |
| **Agent** | 调研、起草、送审、分析反馈、执行修改 | Coddy |
| **评审者 1** | 工程规范评审 | Coding agent（如 Codex、Claude Code） |
| **评审者 2** | 实战经验评审 | 领域专家 |

## 交互节奏

```
决策者:   "这个想法，研究一下"
Agent:    调研→产出初稿→"这里有 Spec V1.0，1000 行，88% 覆盖"
决策者:   "送 coding agent review"
Agent:    Coding agent review→分析反馈→"6 个建议采纳，2 个反对，原因是..."
决策者:   "同意，改"
Agent:    改完→送审→"V1.2，7.4 分"
决策者:   "还不够，再改"
Agent:    再改→...→"V2.8，8.6 分，专家说可以冻结"
决策者:   "名字叫灵感捕手，开工"
```

**每次交互**：
- Agent 先给方案/分析，不等指令
- 决策者确认或修正后 Agent 立即执行
- 外部评审作为第三视角，不替代决策者判断

## 一致性检查清单（交付前必跑）

> **以下为参考示例，根据项目情况增减。** 每个项目有自己特定的一致性关注点——比如有 API 的注意参数一致性，有 UI 的注意交互描述一致性。

### 文字一致性（grep 能查的）

每次送审前，Agent 自行检查：

```
□ 字段计数：全文所有地方的字段数一致
□ SLA 数字：全文所有成功率/目标数字一致
□ 路径：全文所有文件路径/目录名一致
□ 交互模型：全文所有交互描述一致
□ 品牌名：无旧名残留、无 @PLACEHOLDER@ 残留
□ 残留检查：无已删除功能的相关文本
□ 链接：所有外部链接可访问
```

```bash
# 示例：送审前 30 秒自检
grep -c "23字段" spec.md && echo "❌ 还有 23 字段残留（应为 24）"
grep "GPM $" spec.md && echo "❌ 还有 GPM $ 残留"
grep "正在下载.*已用时" spec.md && echo "❌ 还有每阶段推送残留"
grep "@PROJECT@\|@ENG@\|@SLOGAN@" spec.md && echo "❌ 还有 placeholder 未替换"
```

### 设计一致性（grep 查不了的）

送审前人工确认：

```
□ Schema 对齐：Prompt 输出 ⊇ JSON Schema ⊇ 数据模型字段 ⊇ Build Contract 映射表
□ 状态机一致性：状态图 vs 状态转换表 vs 错误处理策略
□ 交互一致性：交互示例 vs 流程描述 vs 定时触发分支
□ 约束一致性：同一字段的 required/type/enum 在三处（Prompt/Schema/DDL）完全相同
```

## @PLACEHOLDER@ 命名规范

> **强制规则。** 从 V0.1 起，所有项目标识用 `@PLACEHOLDER@` 格式。正式名定好后一句 `sed` 全局替换。

### 为什么

想产品名是难事——不该卡住写 Spec 的进度。但如果在 Spec 里硬编码一个临时名字（如"素材工具"），后续品牌化阶段需要手动搜索替换，容易残留、漏改。

`@PLACEHOLDER@` 格式确保：
- **全局唯一**：`@...@` 不是正常中英文会出现的字符组合 → `grep` 零误伤
- **一键替换**：`sed -i 's/@PROJECT@/正式名/g' spec.md`
- **可验证**：替换后 `grep '@' spec.md` 应返回空 → 确认无残留

### 三个标准 Placeholder

| Placeholder | 用途 | 示例 |
|------------|------|------|
| `@PROJECT@` | 中文产品名 | 灵感捕手 |
| `@ENG@` | 英文工程名（repo/package/CLI） | hookly |
| `@SLOGAN@` | 一句话 Slogan | 网络爆款，帮你抓得住，拆得透，做得出 |

### 使用方式

**Phase 1（调研结束）**：决策者给一个简单代称，Agent 填入 placeholder。代称随意取（如"素材工具"），唯一使命是**被替换**。

**Phase 2-5（写 Spec + 迭代）**：全文统一用 `@PROJECT@` / `@ENG@` / `@SLOGAN@`。目录路径、表名、命令示例、代码注释都用这三个占位符。

**Phase 6（品牌化）**：
```bash
sed -i 's/@PROJECT@/灵感捕手/g; s/@ENG@/hookly/g; s/@SLOGAN@/网络爆款，帮你抓得住，拆得透，做得出/g' spec.md
grep '@' spec.md  # 应返回空，确认无残留
```

### 反面教材

```
❌ 硬编码临时名：全文写了 200 次"素材工具"，品牌化时手动替换漏了 3 处
❌ 用拼音首字母："scgj" — 别人看不懂，sed 容易误伤
✅ 用 @PROJECT@ / @ENG@ / @SLOGAN@ — 零误伤，一键替换，可验证
```

## 文档结构模板（附说明）

```markdown
# @PROJECT@ (@ENG@) — 需求规格说明书 VX.X
```

> **说明**：首行即品牌。中文名在前，英文工程名在括号里。版本号从 V1.0 起。
> **必须做**：品牌名和工程名都要有。工程名用于 repo/package/CLI。
> **不要做**：不要叫"XX系统"——太泛。名字从用户视角出发。

```markdown
> @SLOGAN@
>
> **定位** — 受谁启发、不是 1:1 还原、差异化在哪。
> **工程名**：`@ENG@` — repo / package / CLI 统一使用此名。
> 文档结构 — P0 开发只需读 §A + §B + Part 0。
```

> **说明**：抬头要让人 10 秒内知道这是什么、怎么读。
> **必须做**：标注"P0 开发只需读哪几个章节"——减少阅读负担。
> **不要做**：不要写"写给所有人"之类的元说明。直接给信息。

```markdown
## 更新日志
- V1.0 (日期): 初始版本
- V1.1 (日期): 主要改动一句话
- V2.0 (日期): 重大变更
```

> **说明**：每次修改必须记一行。一行一句话，说清楚改了什么。
> **必须做**：日期 + 版本号 + 一句话。
> **不要做**：不要写长段落。一个版本一行。

```markdown
---

## §A 项目背景与价值
### A.1 What — 这是什么？
### A.2 Why — 为什么做？（含举例）
```

> **说明**：§A 是项目级的——任何阶段的人都能看，不需要技术背景。
> **What**：产品是什么、核心卖点、三个阶段三种形态。用日常语言。要举例。
> **Why**：痛点 + 现有方案为什么不行 + 用了之后什么样。必须有一个具体的人物+场景例子（"小王卖咖啡机"）。用 storytelling，不用表格。
> **必须做**：让非技术人 30 秒能看懂、能共情。
> **不要做**：不要用技术术语。不要写"AI 视频素材拆解与复用系统"这种自嗨标题。

```markdown
---

## §B 增量交付路线总览
### B.1 P0/P1/P2 三阶段
### B.2 Artifact 演化一览表
```

> **说明**：§B 是项目级的路线图。读者一眼看到"分几步、每步做什么"。
> **B.1**：表格，P0/P1/P2 各一行。两列：业务视角（功能+价值）+ 技术视角（工具+不引入什么）。
> **B.2**：Artifact 演化表（28 项 × 3 阶段，标 🔵/🟡/⚪）。
> **必须做**：每个阶段写清楚"不引入什么"——和"做什么"一样重要。
> **不要做**：不要写"coding agent 评估提出的 XX 归入 P2"这种评审元信息。这属于评审报告，不属于 Spec。

```markdown
---

## Part 0: P0 — 当前阶段
```

> **说明**：P0 是 MVP。自包含——程序员只读此 Part 即可开工。Part 1/2 是后续扩展，P0 开发时不用读。
> **必须做**：P0 开篇标注"P0 开发只需读 §A + §B + Part 0"。
> **必须做**：所有 artifact（§0.0-§0.N）全部在这一 Part 内。不引用 Part 1/2 的内容。
> **必须做**：验收 Checklist 分两组：硬验收（核心闭环必须过）+ 软验收（P0 增强可后置）。
> **不要做**：不要把 P1/P2 的功能承诺写在 P0 的"做什么"列表里。P0 范围宁窄勿宽。

```markdown
  §0.0 P0 定位
  §0.1 一句话定义
  §0.2 核心成功标准
  §0.3 MVP 范围（做什么 / 不做什么）
  §0.4 用户交互（含进度反馈 + 定时主动反馈）
  §0.5 字段定义（核心字段 + 系统字段，含类型/枚举/必填）
  §0.6 AI Prompt + JSON Schema（prompt 模板 + 机器可校验 Schema）
  §0.7 技术架构图 + 执行流程图（含定时触发分支）
  §0.8 API / 接口规格（命令 + 参数 + 示例）
  §0.9 数据模型（DDL / 建表语句）
  §0.10 状态机（图 + 转换规则表）
  §0.11 错误处理策略（每种失败场景的处理 + 用户看到什么）
  §0.12 数据库策略（各阶段用什么、切换条件）
  §0.13 成本估算（增量成本 + 依赖前提）
  §0.14 部署 / 环境配置（依赖安装 + 目录创建 + 环境变量）
  §0.15 执行顺序（风险优先，非逻辑顺序。Day 0-7）
  §0.16 验收 Checklist（硬验收 + 软验收分两组）
  §0.17 Implementation Contract（字段契约 + Schema 契约 + 去重契约 + 状态契约 + 写入契约）
  §0.18 Build Contract（开工前验证：字段清单 + Schema 映射 + fixture + Day 0 smoke test）
  §0.19 为 P1 预留的演进接口
```

> **说明**：以上为 P0 标准 artifact 清单。顺序建议：先让人理解产品（§0.0-0.4），再给技术细节（§0.5-0.11），最后给工程保障（§0.12-0.19）。
> **必须做**：每个 artifact 都要有——程序员只读这一 Part，不能跳转。
> **必须做**：Implementation Contract 和 Build Contract 是两个独立的东西。前者是"开发时要遵守的约定"，后者是"开工前要验证的东西"。
> **不要做**：不要把反馈闭环做成独立章节。融入用户交互（§0.4）+ 执行流程（§0.7）+ API（§0.8）+ 验收（§0.16）。

## 反模式清单

> 从 hookly 项目中实际踩过的坑。

### 1. V1.0 写太细才送审

**后果**：1250 行返工。方向问题（"大系统蓝图→P0 优先"）应该在骨架阶段发现。

**正确做法**：V0.1 骨架（~300 行）先送审。方向确认后再写全量。

### 2. 评审意见未和决策者逐条确认

**后果**：如果全部照做，P0 会过度工程化（PostgreSQL、队列、审计日志）。

**正确做法**：Agent 列出所有评审意见，每条标注"建议采纳/建议不采纳/需讨论" + 理由。决策者逐条确认。最终决定权在决策者，不在 Agent 也不在评审者。

### 3. 数字在多处不一致

**后果**：23/24 字段、70%/60% SLA 修了 5 轮才对齐。反复送审浪费资源。

**正确做法**：改数字前 `grep` 全文搜所有出现位置，一次性全改。送审前跑一致性检查清单。

### 4. 品牌名拖到最后

**后果**：全文旧名残留难清。

**正确做法**：从 V0.1 就用 `@PROJECT@` / `@ENG@` placeholder。定名后一句 `sed` 全局替换。

### 5. 以为修完了，送审又发现同样问题

**后果**：失去评审者信任，延长迭代轮数。

**正确做法**：送审前用 `grep` 验证，不靠记忆。同一问题出现两次 → 加入检查清单永久化。

### 6. PR 心态替代迭代心态

**后果**：想把一切做完美再送审 → 拖延 → 问题积累 → 返工更大。

**正确做法**：每轮只改当前评审发现的问题，改完立刻送审。不做"顺带优化"。小步快跑。

| 反模式 | 一句话 |
|--------|--------|
| 太细才送审 | 骨架先审，方向对再补细节 |
| 未和决策者逐条确认 | Agent 整理所有意见 + 标注，决策者逐条决定 |
| 数字不一致 | 改前 grep 全文，改后跑检查清单 |
| 名字拖最后 | V0.1 就用 @PLACEHOLDER@ |
| 以为修好了 | 送审前 grep 验证，不靠记忆 |
| PR 心态 | 小步快跑，每轮只改当前问题 |


## 经验教训

1. **外部评审越早越好**——V1.0 就送审，不要等到 V1.5
2. **先写全量再砍**——知道完整版长什么样，才知道 P0 该砍什么
3. **一致性修复是最后 20% 的工作**——也是最容易被忽略的
4. **名字不要最后想**——但也不要最早想。Spec 稳定后、开工前是最佳时机
5. **决策者说"动手"再动**——不在你确认前浪费时间
6. **Agent 必须有立场**——评审意见要过一遍再给决策者，过滤掉过度工程的建议
